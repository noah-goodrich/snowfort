# Project Plan: Snowfort-Audit v0.3.0 — Performance + Quality

*Established: 2026-04-09*

## Objective

Ship a v0.3.0 release that fixes the misleading scoring model (3/100 F on an account
that should score ~63/100 D), eliminates remaining application-side performance
bottlenecks (view-phase parallelization, tag index, single-pass scoring), surfaces
silenced worker errors, and validates everything with a comprehensive smoke test
against a real Snowflake account seeded to trigger violations across all 6 pillars.

## Key Architecture Decisions

### Async: Not Needed
The Snowflake Python connector is sync-only. Wrapping sync calls in
`run_in_executor` adds complexity without benefit — the GIL already allows
I/O concurrency via threads. The real bottleneck is the **view-phase running
sequentially** despite `check_static()` being pure Python. Fix:
`ThreadPoolExecutor`, same pattern already proven in `_execute_parallel`.

### ScanContext Column-Name Access: Deferred to v0.4.0
Converting magic tuple indices across 15+ rule files and 30+ test files is a
large mechanical change that conflicts with the 80/20 rule for this release.
The `tag_refs_index` built in Session 2 demonstrates the pattern.

### SEC_008 SQL Bug: Already Fixed
v0.1.0 had `SHOW GRANTS OF ROLE {role}` without quoting, breaking on a role
named `OPERATIONS`. v0.2.0's N+1 elimination replaced this with batch
ACCOUNT_USAGE queries. Write a regression test to confirm.

## Scoring Model Design

**Problem:** Linear deduction floors at 0 with any volume. 2602 MEDIUM CDC
violations alone produce deduction of 5204, instantly zeroing the score.

**Formula: `10 * ln(1 + raw_deduction)` per pillar, then mean of pillars.**

```python
import math

def _pillar_score(critical: int, high: int, medium: int, low: int) -> float:
    raw = (critical * 10) + (high * 5) + (medium * 2) + (low * 1)
    if raw == 0:
        return 100.0
    dampened = min(100.0, 10.0 * math.log(1 + raw))
    return max(0.0, 100.0 - dampened)
```

Calibration:
- 0 violations → 100/A
- 1 LOW (raw=1) → dampened 6.9 → score 93/A
- 1 CRITICAL (raw=10) → dampened 24.0 → score 76/C
- Security pillar (~165 raw) → score ~49
- Cost pillar (~480 raw) → score ~38
- INCLOUDCOUNSEL overall → mean of pillars ≈ 63/D

Overall score = `round(mean(pillar_scores))` (unchanged from current logic).

## Risks

1. **Scoring model changes break expectations** — Every existing scoring test
   needs updated expectations. Users comparing v0.2 and v0.3 scores see different
   numbers. Mitigation: CHANGELOG explains the change clearly.

2. **View-phase parallelization non-determinism** — ThreadPoolExecutor may
   produce violations in different order. Mitigation: sort final violations by
   rule_id + resource_name for deterministic output.

3. **Smoke test flakiness** — Requires real Snowflake account, network, credits.
   Mitigation: `@pytest.mark.snowflake`, skipped in `make check`, run manually
   as final validation step.

4. **Worker error reporting return type change** — `_run_rules_chunk()` return
   type expands from `(violations, timings)` to `(violations, timings, errors)`.
   Only called by `_execute_parallel`. Low risk, verify all call sites.

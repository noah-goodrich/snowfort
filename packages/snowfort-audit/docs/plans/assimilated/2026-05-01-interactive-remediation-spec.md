# Directive: Interactive Remediation Skill (Cortex-Native)

**Depends on:** Existing `snowfort audit scan --manifest` output (already shipped).
No new rules or infrastructure code required — this directive is pure skill/agent work.

**Priority:** High. The current SKILL.md is a passive recipe. It tells an agent how to
parse output but gives no guidance for conversational querying, challenge mode,
remediation plan building, or deferred/accepted tracking. That gap limits day-to-day
usefulness for anyone who runs the audit and then wants to act on the results.

---

## Objective

Build a Cortex Code skill that drives an **interactive remediation planning session**:
the agent runs the audit, then converses with the user to filter findings, challenge
false positives, build a prioritized remediation plan, and optionally execute SQL fixes
with confirmation. The session ends with a plan the user can commit to version control.

---

## What the Existing Skill Does (Baseline)

`.cortex/skills/snowfort-audit/SKILL.md` covers:

- Run `snowfort audit scan --manifest` → parse JSON findings
- Filter to CRITICAL/HIGH
- Extract `remediation_instruction` for each finding
- Export YAML report

What it **does not** cover:

- Conversational querying ("show me only COST violations over $500 threshold")
- Challenge mode ("is this a false positive? why?")
- Remediation plan builder (prioritize, defer, accept, track)
- Execution loop (apply SQL with confirmation, re-scan to verify)
- Session output (save plan as YAML/JSON for commit)

## Scope

### Phase 1 — Conversational Triage

After the manifest is loaded, the agent enters a conversation loop:

**Supported queries (agent must handle all of these):**

| User intent | Example utterance | Agent behavior |
|-------------|-------------------|----------------|
| Filter by pillar | "show me only Security findings" | Filter manifest by `pillar` |
| Filter by severity | "what's CRITICAL?" | Filter by `severity` |
| Filter by resource | "anything about COMPUTE_WH?" | Filter by `resource` substring |
| Filter by rule ID | "explain COST_001" | Return rule + message + remediation |
| Count / summary | "how many HIGH findings are there?" | Count and group by pillar |
| Worst first | "what should I fix first?" | Sort by severity DESC, then pillar priority |
| Estimated savings | "which cost rules have savings estimates?" | Filter findings whose message contains `$` or `credits` |

The agent must answer these purely from the in-memory manifest — no re-scan unless the
user says "re-scan" or "run again".

### Phase 2 — Challenge Mode

For any finding, the user can say "is this a false positive?" or "why is this flagged?".

The agent must:

1. State the rule rationale (from `rule_name` + `message`).
2. Ask: "Does this apply to your environment? (yes / no / not sure)"
3. If **no**: mark the finding as `accepted_false_positive` and explain how to suppress
   it via `snowfort.toml` exclusions (e.g., `exclude_name_patterns` for COST_012,
   `sso_domains` for SEC_003, etc.).
4. If **not sure**: provide the SQL to investigate (derive from `remediation_instruction`
   context or explain the ACCOUNT_USAGE view the rule queries).

The agent must know (document in the skill) the top 5 most commonly challenged rules
and their standard investigation queries:

| Rule | Common challenge | Investigation SQL |
|------|-----------------|-------------------|
| COST_001 | "We need long suspend for BI burst" | `SHOW PARAMETERS LIKE 'AUTO_SUSPEND' IN WAREHOUSE <name>` |
| SEC_003 | "We use SSO, password policy doesn't apply" | `SHOW NETWORK POLICIES` to confirm one exists |
| COST_012 | "Those are CDC tables, churn is expected" | Check table name against CDC naming patterns |
| OPS_001 | "We use Snowflake Budgets instead" | `SHOW BUDGETS` to confirm budget covers these WHs |
| PERF_006 | "This WH handles burst load, not steady state" | Check `QUERY_HISTORY` P95 vs P50 spread |

### Phase 3 — Remediation Plan Builder

The agent maintains a plan in memory during the session. Commands:

| User intent | Example utterance | Agent action |
|-------------|-------------------|--------------|
| Add to plan | "add this to the plan" | Append finding to `plan.remediate[]` |
| Defer | "defer this for next quarter" | Append to `plan.deferred[]` with reason |
| Accept risk | "accept this, it's by design" | Append to `plan.accepted[]` with reason |
| Show plan | "show the plan so far" | Print current plan state |
| Prioritize | "order by business impact" | Sort `plan.remediate` by severity then estimated savings |
| Save plan | "save the plan" | Write `remediation_plan.yaml` (see schema below) |

**Plan YAML schema:**

```yaml
# remediation_plan.yaml — generated by snowfort interactive session
generated_at: "2026-05-01T09:00:00"
scan_summary:
  total_findings: 42
  critical: 3
  high: 11
  medium: 18
  low: 10

remediate:
  - rule_id: SEC_003
    rule_name: Network Perimeter
    severity: CRITICAL
    resource: Account
    message: "No Account-level Network Policy set"
    remediation_instruction: |
      CREATE NETWORK POLICY corp_network_policy
        ALLOWED_IP_LIST = ('203.0.113.0/24');
      ALTER ACCOUNT SET NETWORK_POLICY = corp_network_policy;
    notes: "Apply after VPN IP range confirmed with IT."

deferred:
  - rule_id: COST_006
    reason: "Warehouse is intentionally oversized for Q2 ETL sprint. Revisit Q3."

accepted:
  - rule_id: COST_001
    resource: BI_WH
    reason: "BI_WH requires 300s suspend for Tableau cache warmth."
```

### Phase 4 — Execution Loop (Optional)

If the user says "apply the plan" or "run the fixes":

1. Show all `remediation_instruction` SQL statements as a numbered list.
2. Ask: "Apply all? (yes / pick numbers / no)"
3. For each confirmed item, execute via `sql_execute` tool.
4. After each execution, show the result and ask: "Mark as done? (yes / retry / skip)"
5. After all items: suggest `snowfort audit scan` to verify the findings are resolved.

**Safety rules the agent must follow:**

- Never execute DDL on `ACCOUNTADMIN`-scoped objects (ACCOUNT, NETWORK POLICY, USER)
  without explicit confirmation: "This affects account-level security. Confirm? (yes/no)"
- Never execute DROP statements without double confirmation.
- Always show the SQL before executing.

### Phase 5 — Session Output

At the end of the session (user says "done", "exit", "save and quit"), the agent:

1. Writes `remediation_plan.yaml` if not already saved.
2. Prints a commit suggestion:
   ```
   git add remediation_plan.yaml
   git commit -m "chore: snowfort audit remediation plan $(date +%Y-%m-%d)"
   ```
3. Prints summary: N items queued for remediation, M deferred, K accepted.

## Key Design Decisions

1. **No new CLI commands.** The manifest JSON from `snowfort audit scan --manifest` is
   the only data source. The skill works entirely from that output. This means the
   interactive skill works with any version of snowfort-audit ≥ 0.4.0.

2. **Agent holds state in memory, not on disk.** The plan is built in-session and only
   written to `remediation_plan.yaml` on save/exit. No intermediate state files.

3. **Challenge mode uses documented investigation queries, not re-scan.** Re-scanning
   for every challenge would be slow and expensive. Investigation queries are lightweight
   `SHOW` commands or narrow `ACCOUNT_USAGE` queries.

4. **Execution is opt-in, not default.** The agent offers to execute after plan review,
   but the default path is plan-and-save only. This prevents accidental DDL.

5. **Separate skill file from SKILL.md.** The existing SKILL.md is a 5-minute passive
   recipe. The interactive skill is a 30-60 minute session. Keeping them separate lets
   users invoke the right one for the job.

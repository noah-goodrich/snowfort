## Context

Explored the full skill delivery mechanism, CLI interface, manifest JSON contract, convention override
system, and the existing two skills. Key findings:

- **Delivery**: Skills live in `.cortex/skills/` in the source tree. Not pip-distributed. Cortex Code
  auto-discovers them when the repo is the working directory.
- **Manifest JSON**: `snowfort audit scan --manifest` outputs `{"violations": [...], "metadata": {...}}`
  where each violation has: `rule_id`, `resource_name`, `message`, `severity`, `pillar`,
  `remediation_instruction`, `remediation_key`, `category`, `context` (rationale), `blast_radius`,
  `quick_win`.
- **Show command**: `snowfort audit show --pillar Security --severity HIGH --rule-id COST_001 --resource COMPUTE_WH`
  provides cached filtering. The skill should use manifest JSON in-memory, not re-invoke `show`.
- **Convention overrides**: Via `pyproject.toml` under `[tool.snowfort.conventions.thresholds]`. No per-rule
  exclusion exists — users adjust thresholds or use `--rule` to filter. The directive spec's reference to
  `snowfort.toml` exclusions is inaccurate; the skill must reference `pyproject.toml` conventions instead.
- **Existing skills**: `snowfort-audit/SKILL.md` (passive recipe) and `snowarch-audit/SKILL.md` (YAML
  frontmatter with tools). Both are thin — ~50-100 lines, no interactive state management.

## Implementation Steps

### Step 1: Create the interactive skill directory and SKILL.md

Create a single file:
```
packages/snowfort-audit/.cortex/skills/snowfort-audit-interactive/SKILL.md
```

The skill file covers all 5 phases from the directive:

1. **Phase 1 — Conversational Triage**: Instructions for parsing `--manifest` JSON, answering
   7 query types (filter by pillar/severity/resource/rule, count, worst-first, savings), all from
   in-memory manifest data — no re-scan.

2. **Phase 2 — Challenge Mode**: Top 10 commonly-challenged rules with investigation SQL:
   - COST_001 (auto_suspend) → `SHOW PARAMETERS LIKE 'AUTO_SUSPEND' IN WAREHOUSE <name>`
   - COST_012 (high churn) → check table naming for CDC patterns
   - SEC_003 (network perimeter) → `SHOW NETWORK POLICIES`
   - SEC_004 (MFA) → `SHOW USERS` to check `HAS_MFA`
   - SEC_007 (zombie users) → `SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY WHERE ...`
   - OPS_001 (resource monitor) → `SHOW RESOURCE MONITORS`
   - OPS_009 (IaC readiness) → `SHOW TAGS IN ACCOUNT`
   - PERF_005 (local spillage) → warehouse sizing query
   - PERF_006 (remote spillage) → `QUERY_HISTORY` P95 vs P50 spread
   - GOV_001 (object comments) → coverage check
   
   For false positive handling: explain `pyproject.toml` threshold overrides (not `snowfort.toml`
   exclusions, which don't exist).

3. **Phase 3 — Remediation Plan Builder**: In-memory plan state with `remediate[]`, `deferred[]`,
   `accepted[]`. YAML schema for `remediation_plan.yaml` output.

4. **Phase 4 — Execution Loop**: Numbered SQL list, selective execution via `sql_execute`, safety
   rules (double-confirm ACCOUNT-scoped DDL, double-confirm DROP, always show SQL first).

5. **Phase 5 — Session Output**: Write `remediation_plan.yaml`, print git commit suggestion, print
   summary counts.

### Step 2: Verify skill invocation

Run `snowfort audit scan --manifest` to get a real manifest, then manually test that the skill
instructions produce correct agent behavior for triage queries, challenge mode, and plan building.
